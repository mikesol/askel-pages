Proposed Skill Taxonomy

Routing model: The top-level playbook skill is a dispatcher. When invoked, it reads context (deal stage, task type, question) and surfaces only the relevant sub-skill. Cron jobs and on-demand tasks that need playbook context load playbook first; it routes them to the appropriate layer. This is the progressive disclosure pattern — don't dump the whole playbook, load what's relevant.

Repo Reorganization

The repo currently serves one master: Steve's working memory. After migration it serves two: Steve's memory and Askel's shared knowledge base (published as a website). That's a meaningful shift in discipline and structure.

The reorganization introduces a cleaner separation across three axes:

  • Process knowledge → .claude/skills/playbook-*/ — how to do things
  • Market knowledge → knowledge/ — what we know about the market
  • Portfolio knowledge → knowledge/portfolio/ — what we know about companies we own

Key Structural Decisions

DecisionRationale
Introduce knowledge/portfolio/ Melers brand assets sit in knowledge/melers/ today, which is fine but doesn't scale. When we acquire company #2, there's no clean place for its brand assets + operational notes. portfolio/ becomes the bucket: one folder per company, with brand/, learnings/, and whatever else accumulates.
Deprecate sourcing-tradecraft Its content is excellent — but it's a private skill and the thesis lives in a sub-reference that cron skills don't cleanly reference. playbook-sourcing absorbs it and becomes the canonical version. The thesis sub-file becomes playbook-thesis.
Thesis as a standalone skill Currently hardcoded in multiple cron skills ("400K–1.5M, profit 100K+, ≤1M ask, 10+ yr Oy"). Any thesis update requires hunting down every reference. playbook-thesis is the single file all crons and sweeps reference.
Don't move knowledge/sourcing/sessions/ That's an archive. It's fine where it is. The thesis reference moves out; the sessions stay.
Knowledge transfer = stub, not a full skill Varia's decision: 1 acquisition is not enough data. The stub exists so the playbook pipeline is complete (no gaps in the stage sequence), but it points to proto-knowledge rather than pretending to be a playbook. Expected to get filled out after acquisitions 2–3.

Auth Gating — What's Needed

Cloudflare Zero Trust (Access)

This is the right tool: creates a login gate in front of the Pages site, requires authentication with a team email (@askelventures.com). The current Cloudflare API token doesn't have Access: Apps and Policies permission.

Option A (recommended): In the CF dashboard → My Profile → API Tokens → Create Token → "Edit Cloudflare Workers" template → add Access: Apps and Policies (Edit) permission → save token → Steve can automate from there.

Option B: Set it up manually in the CF dashboard → Zero Trust → Access → Applications → Add an application → select the Pages subdomain → create a policy matching @askelventures.com email.

Current state of this site

Deployed without auth gating. It's analysis content, not sensitive deal data, so the risk is low — but it shouldn't stay public indefinitely. Once one of the above options is done, Steve can wire up the gate programmatically.

What Stays the Same

  • Cron jobs — same schedule, same channels. Content improves because they now follow precise playbook checklists.
  • knowledge/sourcing/sessions/ — sourcing session archive stays exactly where it is.
  • Trello workflow — prospecting → Trello card creation is unchanged. playbook-prospecting just gives Steve more precise instructions for when a card is warranted.
  • CLAUDE.md — will be updated to reference the new playbook skills in the knowledge base index, but its structure stays the same.
  • All non-playbook skills — trello-cli, finder-fi, google-workspace, askel-pages, etc. — untouched.
